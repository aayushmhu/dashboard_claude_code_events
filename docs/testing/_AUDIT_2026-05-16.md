## Key Finding: CEO's hypothesis was partially correct but misattributed

The CEO brief stated "opus-trivial-tools and opus-on-research-tasks return 0 results because they query model on PostToolUse rows where it's NULL."

**This is not accurate.** Both rules query model on `Stop`/`SubagentStop` rows, where model IS populated (762 Opus rows, 0 null). The SQL itself is correct.

The actual explanation for the zero result in the prior dry-run:

- **opus-trivial-tools** (turn_count=2 in current run, was 0 in prior run): The rule fired 0 at the time because there were fewer qualifying sessions in the 30-day window at that moment. Today there are 2 turns — below the threshold of 5. Not a bug.
- **opus-on-research-tasks** (session_count=2 now, was 0 in prior run): The prior dry-run may have been run at a different time when sessions were outside the 30-day window. Today it fires with session_count=2 = threshold (2).

The prior dry-run that showed 0 for both may have been using a different window or the "model on PostToolUse" theory may have been tested incorrectly by querying PostToolUse events directly rather than checking what the rules actually JOIN to.

**The SQL is correct. No rule rewrites are needed.**
